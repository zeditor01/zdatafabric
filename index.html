3 Patterns for a Data Fabric for Mainframe Data

  • Z Data in Motion
  • Z Data at REST
  • ERIS

The diagram below depicts a very common approach for building a Data Fabric that contains data from IBM Z Systems of Record. The idea is to stream operational data out of the System of Record as soon as it changes, and feed it into a digital integration hub (on-cloud or on-prem) for curation of data and creation of data service APIs to support business applications on the cloud.

This approach works well for applications of limited scope, where the domain of data needed to support the business applications is clearly defined and the data provisioning infrastructure can be contained to manageable size and complexity. For example, portals for personal banking mobile apps often adopt this pattern.

The core data enablng technology for this pattern is data replication and/or event processing software. The data movement technology itself is conceptually simple, but it carries a large amount of technical debt when you consider the devops lifecycle factors for data management including authentication, encryption, ci-cd pipelines, change control, observability, automated operations, continuous service levels, high availability, disaster recovery and so forth.

If you scale the cost, technical debt and timescales of extending the data provisioning infrastructure to include all the other data domains, it may be attractive to consider alternative/additional Data Fabric patterns to address the breadth of data that you want to consume within your cloud applications. The "Z Data at REST" and "ERIS"sections describe alternative approaches that reduce or avoid the extensive copying of data to other platforms. The ideal data fabric may be a combination of several patterns.

This section provides worked example of three of the core data replication and event publish patterns that support the "Z Data in Motion" pattern.

CDC Replication is aimed at heterogenous data replication, and streams changed data over TCPIP sockets.

  • It supports all the regular relational databases ( Db2 Oracle, SQL Server etc... ) as sources and targets.
  • Additionally, it supports non-relational databases such as IMS and VSAM.
  • And it supports non-relational targets such as Apache Kafka (various distributions of Kafka).

Q Replication is primarily aimed at homogeneous DB2 Replication, and streams changed data over Websphere MQ links.

  • It supports in-line DDL Replication between Db2 z/OS sources and Db2 z/OS targets..
  • It supports publishing to MQ in open formats, enabling a pub/sub integration model with MQ consumers.
  • It is the basis for the GDPS Active-Active continuous availability solution.

Datagate is a lightning fast data replication protocol between a Db2 z/OS V12 source and a Db2 Software Container, such as IDAA or Datagate.

  • It supports a choice of conventional row-based Db2 targets, or columnar Db2 targets that are optimised for Analytics.
  • It supports acceleration of analytical SQL queries on Db2 z/OS by transparently routing them to a columnar Db2 Software Container.
  • It provides the ability to place your operational Db2 z/OS data in the cloud to support new cloud-native development projects.
