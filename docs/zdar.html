3 Patterns for a Data Fabric for Mainframe Data

  • Z Data in Motion
  • Z Data at REST
  • ERIS

The "Z Data at REST" pattern is motivated by avoiding the costs, complexity and timescales of moving the data off Z, and making use of the wide range of data service solutions that exist to serve data direcly from z/OS. Cloudpak for Data supports a data virtualisation layer that accesses data wherever it is, so that you don't need to build and maintain constant data movement processes to maintain a copy of the mainframe data.

The "Z Data at REST" pattern works well with Cloudpak for Data, but it also works well with ANY application that is able to consume industry-standard APIs, such as REST, SQL, ODBC, JDBC and so on.

The worked examples in this section show how z/OS data resources can be surfaced using a range of APIs (REST, SQL, ODBC, JDBC etc....) for consumption by cloud applications directly, or by Data and AI integration solutions like Cloudpak for Data.

DB2 DDF is a standard component of Db2 z/OS, that supports both REST APIs and SQL-based APIs.

  • It can serve REST APIs based on calls to SQL and Stored Procedures.
  • It can serve a variety of SQL-based APIs ( JDBC, ODBC etc... ).
  • It is tightly integrated into the Db2 z/OS service (Data Sharing Availability, Reliability, Performance, Secuirty etc...)

DVM provides herterogeneous data connectivity for data sources like IMS, VSAM, Syslogs, Adabas, Datacom etc...

  • It provides SQL access to heterogeneous data sources, both locally on z/OS, and remotely from LUW clients.
  • It can be used in conjunction with z/OS Connect to provide REST APIs to access it's range of heterogeneous data sources.
  • It includes a DRDA Application Server, which allows heterogenous data sources to be consumed by Db2 using constructs like Table Functions and Aliases.

QMF (the original green-screen reporting tool from the early days of Db2 z/OS) contains a treasure trove of data integration capabilities.

  • Modern LUW and browser dashboard development.
  • QMF Data Service to develop results sets as callable objects.
  • QMF Server (zIIP eligible) to serve up query results sets by REST and SQL APIs.
